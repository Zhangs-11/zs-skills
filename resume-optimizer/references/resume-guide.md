# 简历写作规范

## 定位：简历是广告，不是档案

简历的唯一目标是换来一次面试，不是完整记录你的职业生涯。

档案追求全，广告追求打中。筛选一份简历的时间通常只有几十秒，能被记住的只有两三个点——所以简历的每一行都要回答同一个问题：**这一行能不能让筛选者多看你一眼？** 不能的就删掉。

由此推出三条贯穿全文的原则：

1. **具体压倒抽象**：「学习能力强」是零信息，「两周内从零上手 Go 并独立交付一个线上服务」才有说服力。
2. **成果压倒动作**：「负责后台开发」是动作，「重构任务调度模块，任务失败率从 5% 降到 0.3%」是成果。
3. **能量化就量化**：性能提升多少、覆盖多少用户、节省多少人力，数字自带可信度。

## 简历骨架

标准结构与默认顺序：

| 顺序 | 模块 | 说明 |
|------|------|------|
| 1 | 个人信息 | 姓名、电话、邮箱等 |
| 2 | 教育经历 | 学校、学历、专业、起止时间 |
| 3 | 个人优势 | 2～3 条差异化亮点 |
| 4 | 专业技能 | 技术栈关键词，按经验分层 |
| 5 | 工作经历 | 时间倒序 |
| 6 | 项目经验 | 精选，不求全 |

两条结构性要求：

- **工作经历和项目经验分开写**。不是每段工作都值得配项目，早年的项目没有参考价值就别占篇幅。
- 六个模块齐全才算一份完整简历，缺了要补。

特殊情况调整：

- **最高学历为大专**：教育经历移到简历最后，先让能力说话。
- **专升本 / 成人本科**：不单列教育经历模块，改为在个人信息里写「学历：本科」和「工作经验：X 年」。

## 个人信息

必写四项：姓名、性别、电话、邮箱。

以下内容一律不写：期望薪资、是否在职、婚姻状况、籍贯、政治面貌。年龄也可以省——学历加毕业年份足够推算。

博客、GitHub 链接只在长期维护、内容拿得出手时才放；一个三年没更新的 GitHub 是减分项。

## 教育经历

写清学校、学历、专业、入学和毕业时间。奖学金、竞赛获奖、有含金量的证书可以附在这里。

## 个人优势

个人优势的判定标准只有一条：**同龄同岗的人大多不具备**。人人都会的框架、人人都过的流程，写上去等于没写。

每条都要经得起两个检验：是真事、够具体。写 2～3 条即可，参考方向：

- 技术面覆盖广：既有 C 端高并发经验、又完整做过 B 端权限体系这类跨场景经历
- 特定生态的深耕：如支付、音视频、小程序生态的二次开发经验
- 完整项目周期经验：从方案设计、技术选型到上线运维全程参与过大型项目
- 可验证的社区影响力：开源项目 Star 数、技术文章阅读量、演讲分享经历

反例改写示范——不要写「沟通能力强，抗压能力好」，改成「作为跨三个团队的接口人，推动 xx 项目在两周内完成联调上线」。

## 专业技能

第一轮筛简历的常常是 HR，不懂技术、只认关键词。所以这个模块要覆盖目标岗位 JD 上的常见技术词：语言、框架、数据库、中间件、工程化工具，以及 AI 编程工具。

按工作年限分层，写的东西完全不同：

- **3 年以上**：删掉 Git、基础 CSS 这类入门技能，转而体现框架原理、源码阅读、周边生态。
- **5 年以上**：要出现系统设计、架构选型、带人做项目的字眼，且体现技术广度而非单一框架。
- **8 年以上**：要体现某个业务领域的完整解决方案能力，往业务专家方向写。

两个加分方向：

- **AI 编程**：主流 AI 编程工具的实际使用经验、AI 辅助下的开发流程改造，现在是显性加分项。
- **全栈**：前端补一点后端能力、后端补一点前端能力，匹配现在团队对全栈的偏好。

措辞红线：禁止出现「了解 xxx」。了解等于不会，要么写「熟悉」并做好被追问的准备，要么删掉。

## 工作经历

时间倒序排列。每段写：公司、职位、起止时间、工作职责。

职责分条写，每条落在成果上。「参与后端开发」这种话面试官一天看八十遍，换成「主导 xx 服务拆分，接口平均耗时下降 40%」。

两类风险提前准备好口径，而不是在简历上遮掩：

- 跳槽频繁：想清楚每次变动的合理叙事。
- 空窗期较长：准备好真实且体面的解释。

## 项目经验

### 选哪些项目

项目经验是能力的证据链，不是工作日志。选择策略：

- 应届生 2～3 个，社招最多 5 个。
- **第一个项目**是门面：规模最大、最近两年、你打算在面试中主讲的那个。
- **第二个项目**做补充：最好换一个技术栈或业务形态，证明广度。
- 其余项目一两行带过即可，不展开。

优先级从高到低：公司线上项目 > 实习项目 > 学校真实项目。个人练手项目和教程项目尽量不写。

### 单个项目的格式

```text
**项目名称**（起止时间）｜ 你的角色
项目介绍：一句话说清这是什么行业的什么产品，有哪些核心功能，你负责哪块。
技术栈：整个项目的核心技术栈（不只是你用到的部分）
项目职责：
  - 4～7 条，每条格式：用什么技术，做了什么事，解决什么问题，带来什么效果
项目成果：
  - 3～4 条，写结果不写功能
```

前两个项目必须写项目成果，后面的可以省。

各字段的具体要求：

- **项目名称**：写真名，具体到客户或产品。「xx 银行对公开户审批系统」远比「后台管理系统」可信。
- **角色**：5 年以上经验必须体现负责人角色——项目负责人、端负责人或模块负责人。
- **项目介绍**：让一个不认识你的人 10 秒内明白行业、产品目标和你的位置，否则后面的技术细节无从聊起。
- **项目职责**：拒绝功能清单。每条都套「技术 + 难点 + 效果」，5 年以上要出现架构和系统设计层面的条目。示例：
  - 设计基于消息队列的订单异步化方案，削峰后大促期间零故障，峰值吞吐提升 3 倍。
  - 搭建前端监控体系，接入错误上报与性能采集，线上白屏问题定位时间从小时级降到分钟级。
  - 引入 AI 代码评审流程，结合 CI 卡点，团队线上缺陷率环比下降 25%。
- **项目成果**：只写结果。用户量、收入、效率、稳定性，挑最硬的 3～4 条。

## 评审简历时的检查清单

拿到一份现成简历做评审，按以下顺序过一遍：

1. 六个模块是否齐全，顺序是否合理（注意学历特殊情况的调整规则）。
2. 有没有该删的信息（期望薪资、婚姻状况、荒废的 GitHub 等）。
3. 个人优势是不是真的「优势」，还是人人都有的技能。
4. 专业技能与工作年限是否匹配，有没有「了解 xxx」。
5. 工作职责是动作还是成果，能量化的是否量化了。
6. 项目选择是否得当，第一个项目扛不扛得起主讲。
7. 每一条内容在面试中被追问时，候选人答不答得上来——答不上来的建议删除或改写。
